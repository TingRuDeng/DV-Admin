# DV-Admin 文档同步检查清单

> 本文档定义代码变更时需要同步更新的文档。提交 PR 前请确保完成相关检查。

---

## 检查清单使用方法

1. 根据代码变更类型，找到对应的检查项
2. 逐项检查是否需要更新文档
3. 在 PR 描述中说明文档更新情况
4. 如果确定"无文档影响"，请在 PR 中说明原因

---

## API 变更检查

### 新增 API

- [ ] 更新 `docs/API_ENDPOINTS.md`
  - [ ] 添加端点说明
  - [ ] 添加请求参数说明
  - [ ] 添加响应格式说明
  - [ ] 添加认证要求
- [ ] 检查是否需要更新 `docs/ARCHITECTURE.md`
  - [ ] 是否涉及新的认证流程？
  - [ ] 是否涉及新的权限模型？
- [ ] 检查是否需要更新前端 API 调用
  - [ ] 在 `frontend/src/api/` 添加调用方法

### 修改 API

- [ ] 更新 `docs/API_ENDPOINTS.md`
  - [ ] 更新请求参数说明
  - [ ] 更新响应格式说明
- [ ] 检查前端调用是否需要同步修改
  - [ ] 搜索调用点：`grep -r "api/xxx" frontend/src/`
- [ ] 如本次修改影响共享 API 契约，确认两个后端（Django + FastAPI）都已修改

### 删除 API

- [ ] 更新 `docs/API_ENDPOINTS.md`
  - [ ] 移除相关端点说明
- [ ] 检查前端是否有调用
  - [ ] 移除前端调用代码
- [ ] 如该接口属于共享契约，确认两个后端都已删除

## 文档更新流程

1. **确定变更类型** - 根据代码变更确定文档更新范围
2. **检查现有文档** - 阅读相关文档，确定更新位置
3. **更新文档** - 保持文档风格一致
4. **验证准确性** - 确保文档与代码一致
5. **提交变更** - 在 PR 中说明文档更新

---

## 文档风格指南

### 保持一致性

- 使用相同的术语
- 保持格式统一
- 遵循现有结构

### 保持准确性

- 描述实际行为，不是"应该怎样"
- 代码与文档冲突时，以代码为准
- 及时更新过时信息

### 保持简洁

- 避免重复
- 引用权威文档，不复制内容
- 使用表格和列表提高可读性
