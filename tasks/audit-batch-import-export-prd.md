# 审计日志、批量操作与导入导出状态 PRD

## 背景

当前系统已具备基础能力，但仍偏同步和一次性反馈：

- 审计日志页 `frontend/src/views/system/log/index.vue` 已提供分页、操作人/方法/状态/时间筛选和详情弹窗，但 `requestId` 尚未持久化，跨日志上下文追踪仍不完整。
- Django 与 FastAPI 已提供 `/api/v1/system/logs/*` 查询、统计和删除接口，后端测试覆盖见 `backend/drf_admin/apps/system/test_logs.py` 与 `fastapi/tests/test_logs.py`。
- 用户导入弹窗 `frontend/src/views/system/user/components/UserImport.vue` 以单次上传结果提示为主，缺少任务进度、历史记录和失败明细复查。
- 通用 CURD 导入导出能力位于 `frontend/src/components/CURD/usePageContentFileActions.ts`，当前主要处理本地导出、远程导出调用和同步导入回调。
- 用户、角色、通知等页面已有批量删除入口，但缺少统一任务状态、部分失败明细和可重试机制。

## 目标

- 让管理员能追踪一次后台操作的发起人、对象、结果、耗时和失败原因。
- 让批量删除、批量导入、批量导出从“同步按钮反馈”升级为“可查询任务”。
- 让导入导出具备进度、历史、失败明细下载和重试入口。
- 保持 Django 与 FastAPI 的对外契约一致，前端不感知后端实现差异。

## 非目标

- 本 PRD 不直接实现代码。
- 不在没有产品确认前新增数据库表或异步队列依赖。
- 不改变现有 `/api/v1/system/logs/*` 的基础分页契约。
- 不要求第一期支持所有业务模块，先覆盖用户、角色、通知三个高频批量场景。

## 当前实施状态

2026-07-18 已完成第一期的日志只读体验第一小步：

- Django 与 FastAPI 新增 `GET /api/v1/system/logs/{id}`，沿用 `system:logs:query`、日志数据范围和敏感字段脱敏规则。
- 前端日志页补充操作人、请求方法、执行状态筛选，以及状态、响应码和详情弹窗。
- 双后端运行时测试覆盖详情权限、404、数据范围和字段脱敏，前端 E2E 覆盖失败日志详情链路。
- 详情端点已进入 API、字段和前端字段契约守卫。

以下内容仍未实施：

- `requestId`、对象类型和对象 ID 的持久化、筛选与展示。
- 中间件对 `errorMsg/responseBody` 的稳定采集策略。
- 统一任务状态契约、用户导入结果组件和批量任务化。

## 用户故事

- 作为管理员，我希望查看一条操作日志的完整请求上下文，以便定位失败原因。
- 作为管理员，我希望批量删除时知道哪些记录成功、哪些失败，以及失败原因。
- 作为管理员，我希望导入用户后能查看失败行、字段错误和可下载的失败明细。
- 作为管理员，我希望导出大数据量时看到任务状态，而不是等待浏览器请求长期阻塞。
- 作为维护者，我希望任务、日志和导入导出结果有统一契约，避免每个页面各写一套状态处理。

## 功能范围

### 1. 审计日志 UX

- 日志列表增加操作结果、HTTP 状态、请求 ID、对象类型和对象 ID 字段。
- 日志详情抽屉展示请求路径、方法、参数摘要、响应摘要、耗时、IP、浏览器、操作系统和错误堆栈摘要。
- 支持按操作人、方法、状态、对象类型、请求 ID、时间范围筛选。
- 删除日志继续保留权限 `system:logs:delete`，查询详情沿用 `system:logs:query`。

### 2. 批量任务状态

- 定义统一任务状态：`pending`、`running`、`succeeded`、`partial_failed`、`failed`、`canceled`。
- 定义统一任务进度：`totalCount`、`successCount`、`failedCount`、`processedCount`。
- 任务详情记录每条失败项的业务主键、展示名称、失败码和失败原因。
- 批量删除先覆盖用户、角色、通知；后续再扩展字典、菜单、部门。

### 3. 导入导出状态

- 导入任务记录文件名、文件大小、发起人、开始时间、结束时间、状态和失败明细下载地址。
- 导出任务记录筛选条件摘要、字段集合、文件地址、过期时间和生成耗时。
- 用户导入第一期继续允许同步完成，但接口返回结构需能映射为任务结果。
- 大文件或长耗时导出进入异步任务，不阻塞页面请求。

## 数据契约草案

### TaskSummary

- `id`: 任务 ID
- `type`: `batch_delete` / `import` / `export`
- `module`: 业务模块，如 `users`、`roles`、`notices`
- `status`: 统一任务状态
- `totalCount`: 总数
- `successCount`: 成功数
- `failedCount`: 失败数
- `processedCount`: 已处理数
- `createdBy`: 发起人
- `createdAt`: 创建时间
- `updatedAt`: 更新时间

### TaskFailure

- `taskId`: 任务 ID
- `rowNumber`: 导入行号，可为空
- `objectId`: 业务对象 ID，可为空
- `objectName`: 业务对象名称，可为空
- `errorCode`: 失败码
- `message`: 失败原因

## 分期计划

### 第一期：只读体验与契约

- [x] 补充日志详情接口和前端详情弹窗。
- [ ] 持久化 `requestId` 与错误摘要，并补充对应筛选和展示。
- [ ] 定义任务状态契约和前端类型，不改变现有批量操作行为。
- [ ] 用户导入结果映射到统一任务结果展示组件。

### 第二期：批量删除任务化

- 用户、角色、通知批量删除返回任务结果。
- 前端批量操作展示成功、失败和可重试明细。
- 后端记录每条失败项，不再只返回通用失败消息。

### 第三期：导入导出异步化

- 导入导出创建任务，提供任务列表、详情、失败明细下载和导出文件下载。
- 支持任务重试和过期文件清理。
- 补充双后端契约测试与前端行为测试。

## 验收标准

- 审计日志详情能定位一次失败操作的请求 ID、用户、路径、状态和错误摘要；当前已覆盖用户、路径和状态，`requestId` 与错误摘要采集仍待后续迁移。
- 批量删除 10 条数据时，如果其中 2 条失败，页面明确显示 8 成功、2 失败和失败原因。
- 用户导入失败行可在页面查看，并可下载失败明细。
- 导出任务可查看状态、下载文件，并在文件过期后给出明确提示。
- Django 与 FastAPI 对外字段、状态枚举和错误码由契约测试锁定。

## 待确认问题

- 是否引入异步队列；如不引入，第一期使用数据库任务表加同步执行。
- 任务历史保留时长和文件过期时间。
- 审计日志是否保存请求参数原文，还是只保存脱敏摘要。
- 是否为双后端 `OperationLog` 增加 `requestId` 字段，并如何迁移已有数据库。
- 错误摘要从统一错误包裹提取还是由业务端显式写入。
- 批量任务失败项是否允许逐条重试，还是只支持整批重试。
