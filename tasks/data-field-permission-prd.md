# 数据权限与字段权限 PRD

## 背景

当前 DV-Admin 已具备接口级、菜单级和按钮级 RBAC：

- Django：`backend/drf_admin/utils/permissions.py` 的 `RBACPermission` 基于 `required_permissions` 做接口级校验。
- FastAPI：`fastapi/app/api/deps.py` 的 `PermissionChecker` 基于 `require_permissions` 做接口级校验。
- 前端：角色管理已支持 `dataScope/deptIds`，并与 Django、FastAPI 的角色契约保持一致。

对标成熟后台系统后，项目已完成数据范围与字段级访问控制的 v1 最小闭环。本文件继续作为当前权限语义、覆盖边界和后续扩展条件的产品依据。

## 当前状态与 v1 决策

2026-07-18 确认权限能力 v1 在以下高价值对象内闭环：

- 数据范围：用户列表、操作日志列表和后台通知管理。
- 字段读取：用户 `mobile/email`、操作日志 `requestBody/responseBody/ip`、通知 `targetUserIds/content`。
- 字段写入：后台用户 `mobile/email`、后台通知 `targetUserIds`。
- 治理守卫：角色 `dataScope/deptIds` 契约、字段权限码目录和双后端运行时测试。

v1 之后不按资源数量继续铺开：

- 角色、菜单、部门和字典属于平台配置对象，当前不接数据范围或字段权限。
- 个人中心资料维护采用“本人可编辑字段”语义，不直接复用后台角色字段写权限。
- 新对象只有出现明确的敏感字段、部门归属或越权场景时，才进入权限扩展评审。

## 目标

1. 为角色增加明确的数据范围配置，并由后端查询层强制执行。
2. 为高敏感字段提供读取隐藏/脱敏与写入拒绝能力。
3. Django 与 FastAPI 保持同一外部契约，前端不感知后端实现差异。
4. 保持接口级 RBAC 作为第一层校验，数据权限和字段权限作为第二层收敛。

## 非目标

- 本次 v1 边界确认不继续修改数据库模型或业务代码。
- 不用前端隐藏列替代后端强制权限。
- 不为所有模型接入字段权限，不把平台配置对象机械纳入数据范围。
- 不引入临时 fallback 或绕过权限失败的静默降级。

## 数据权限模型

建议角色数据范围枚举：

| 值 | 名称 | 语义 |
|---:|------|------|
| 1 | 全部数据 | 不追加数据范围过滤 |
| 2 | 本人数据 | 仅允许访问 `owner/user_id/created_by` 指向当前用户的数据 |
| 3 | 本部门数据 | 允许访问当前用户所属部门数据 |
| 4 | 本部门及子部门数据 | 允许访问当前部门树下所有部门数据 |
| 5 | 自定义部门数据 | 允许访问角色绑定的部门集合 |

多角色合并规则建议采用“权限并集”：用户拥有多个角色时，取可访问部门集合并集；如任一角色拥有全部数据，则视为全部数据。

## 字段权限模型

建议字段策略：

- `readable=false`：响应中移除字段。
- `masked=true`：响应中保留字段但按字段类型脱敏。
- `writable=false`：写接口拒绝提交该字段，返回 403 或 400。

字段权限 v1 覆盖：

- 用户：`mobile`、`email`。
- 操作日志：`requestBody`、`responseBody`、`ip`。
- 通知：`targetUserIds`、`content`。

当前采用显式权限码控制上述字段，不引入可动态配置任意模型字段的通用权限表。

## 后端执行边界

后端必须作为唯一安全边界：

1. 接口权限先执行，失败直接拒绝。
2. 数据权限在 queryset/query builder 层追加过滤，不能只在 serializer/schema 后过滤。
3. 字段读取权限在 serializer/schema 输出前处理。
4. 字段写入权限在 request validation 后、业务写入前处理。
5. 超级管理员可按产品决策选择绕过数据范围，但仍应记录审计日志。

## 前端交互

角色表单当前已提供：

- 数据范围下拉框。
- 自定义部门树，仅在数据范围为“自定义部门数据”时展示。
- 字段权限通过菜单/按钮权限树中的显式权限码授予，暂不提供通用字段策略编辑器。

前端权限控制只用于交互提示和减少误操作，不作为安全边界。

## 契约与测试

v1 已建立以下守卫：

- `scripts/api_field_contracts.py` 登记角色 `dataScope/deptIds` 字段。
- `scripts/field_permission_contracts.py` 登记字段原文和写入权限码，并校验运行时代码、初始权限树与测试 fixture。
- Django/FastAPI 运行时测试覆盖数据范围越权、多角色合并、字段脱敏和字段写入拒绝。
- 前端契约与单元测试覆盖角色数据范围字段和高价值对象响应类型。

## 分阶段实施

### 阶段 1：数据范围最小闭环（已完成）

- 给角色增加 `dataScope`。
- 给角色增加自定义部门关系。
- 用户列表、日志列表和后台通知管理接入数据范围过滤。
- 补双后端运行时越权测试。

### 阶段 2：字段读取权限（已完成）

- 建立字段权限目录。
- 用户、日志和后台通知接入字段脱敏。
- 补字段响应契约和前端展示测试。

### 阶段 3：字段写入权限（v1 已完成）

- 写接口增加字段写入拒绝。
- 后台用户 `mobile/email` 与后台通知 `targetUserIds` 写入接入权限校验。
- 角色和个人中心资料维护不在 v1 字段写入范围内。
- 补越权写入测试。

### 阶段 4：治理自动化（已完成）

- 将数据权限/字段权限接入契约报告。
- 在文档中标注已覆盖资源和暂未覆盖资源。

## 后续扩展条件

后续仅在满足以下任一条件时扩展权限覆盖：

- 资源存在明确的部门、创建人或负责人归属，可以定义稳定的数据范围语义。
- 字段包含个人信息、凭据、请求原文或其他需要脱敏的敏感内容。
- 已出现真实越权风险、合规要求或可复现的产品需求。
- Django、FastAPI、前端、契约和迁移可以在同一阶段完成闭环。

## 风险

- 新资源的数据范围语义仍需单独确认，不能直接套用现有对象的归属字段。
- 新敏感字段的脱敏策略需要统一，否则会造成前后端展示不一致。
- 双后端模型和迁移必须同步，否则会重新引入能力漂移。

## 验收标准

- v1 覆盖对象的数据范围由后端强制，不能通过手写请求绕过。
- v1 覆盖字段的读取和写入权限由后端强制，不能通过前端 DevTools 绕过。
- Django 与 FastAPI 对同一用户、同一角色配置和 v1 覆盖对象返回一致结果。
- API 契约、字段契约、模型契约和运行时测试均通过。
